# TODO: Анализ ROSH по про-игрокам

## Уточненная постановка

Глобальная задача: расширить текущий ROSH-анализ по героям анализом по игрокам, но только для про-игроков.

Для каждого слота выбора героя нужно предусмотреть возможность указать про-игрока, найти его через STRATZ, получить `steamAccountId`, затем загрузить player+hero статистику и учесть ее в итоговом расчете вероятности победы Radiant или Dire.

При этом player-specific анализ не должен быть обязательным режимом. Необходимо добавить чекбокс `Учитывать героев`, который фактически переключает расширенный режим расчета:

- если чекбокс выключен, форма и вычисления работают строго по старому hero-only сценарию;
- если чекбокс включен, в интерфейсе появляются поля для ввода игроков, а расчет начинает учитывать player-specific данные;
- без включенного чекбокса никакие player-specific данные не должны влиять на итоговую модель;
- старый формат должен оставаться полностью рабочим и совместимым с текущим поведением.

## Важные ограничения и подтвержденные выводы по STRATZ

- Для этой задачи ключевой идентификатор в STRATZ — это `steamAccountId`, а не `userId`.
- Поиск по нику возможен через `stratz.search(request)`.
- Так как нас интересуют только про-игроки, при поиске нужно использовать `searchType: [PRO_PLAYERS]`.
- Поиск про-игроков может возвращать несколько кандидатов, поэтому нельзя автоматически брать первый результат.
- У найденного результата полезно сохранять не только `steamAccountId`, но и `name`, `proSteamAccount.name`, `aliases`, `team`, `isAnonymous`, `isStratzPublic`.
- Для получения player+hero агрегатов STRATZ уже дает специализированный запрос `plus.playerHeroHighlight(steamAccountId, heroId)`.
- Для расширенной статистики доступны `player.heroPerformance(heroId, request)` и `player.matches(request)`.
- Запрос `plus.draft(...)` использовать нельзя.
- Текущий `page.rosh.stats(...)` слишком узкий и не решает задачу анализа по конкретным игрокам, поэтому player-specific часть нужно строить отдельно и затем подмешивать в текущую модель ROSH.
- Для anonymous-аккаунтов player-specific статистика может быть недоступна: это нужно считать обязательным fallback-сценарием.

## Предпочтительный путь данных в STRATZ

1. Поиск про-игрока по нику через `stratz.search(request)` с `searchType: [PRO_PLAYERS]`.
2. Выбор конкретного кандидата и сохранение его `steamAccountId`.
3. Получение быстрой агрегированной статистики player+hero через `plus.playerHeroHighlight(steamAccountId, heroId)`.
4. При необходимости получение более детальных агрегатов через `player.heroPerformance(heroId, request)`.
5. При необходимости получение списка реальных матчей на герое через `player.matches(request)` с фильтром `heroIds`.
6. Передача подготовленного блока player+hero метрик в итоговый расчет ROSH, но только если включен режим расширенного анализа.

## Функциональные режимы

### 1. Базовый режим

- Чекбокс `Учитывать героев` выключен.
- Интерфейс показывает только текущие hero-поля.
- Payload и серверный контракт остаются максимально близкими к текущему формату.
- Итоговый расчет работает строго по старой hero-only логике без учета игроков.

### 2. Расширенный режим

- Чекбокс `Учитывать героев` включен.
- Для каждого слота дополнительно отображается поле ввода про-игрока.
- После выбора игрока система получает `steamAccountId` и player+hero статистику.
- Итоговый расчет учитывает и героев, и player-specific метрики.

### 3. Fallback внутри расширенного режима

- Если чекбокс включен, но игрок не найден, player-specific данные для этого слота не должны ломать весь расчет.
- Если игрок найден, но статистика недоступна, слот должен откатываться в hero-only вклад.
- Если STRATZ временно недоступен, общий расчет не должен разваливаться.

- Основным быстрым запросом сделать `plus.playerHeroHighlight(steamAccountId, heroId)`.

- Если нужен слой верификации или более глубокая аналитика, получать реальные матчи через `player.matches(request)` с фильтром `heroIds`.

### 7. Изменение серверного контракта и слоев интеграции

- Расширить контракт анализа от структуры "список героев" к структуре "slot -> hero + optional pro player".

- В старом режиме продолжать принимать и обрабатывать текущий hero-only payload без обязательных player-полей.
- В расширенном режиме принимать player-specific поля, `steamAccountId` и связанные метаданные слота.
- Явно разделить быстрые запросы на поиск кандидата и более тяжелые запросы на загрузку player+hero статистики.
- Продумать кэширование player+hero данных, чтобы не дергать STRATZ повторно при одинаковых комбинациях.

- Определить минимальный набор player-specific метрик, которые действительно полезны для ROSH: winrate на герое, число матчей, свежесть практики, недавний IMP, роль и, возможно, KDA-профиль.
- Разделить метрики на базовые и второстепенные, чтобы сначала внедрить устойчивую версию модели.
- Добавить понятие надежности данных: малая выборка не должна влиять на итог сильнее, чем большая.
- Решить, какой вес давать свежей форме за последний месяц относительно общей истории на герое.

- Применять этот вклад только если чекбокс включен.

### 9. Fallback-логика

- Если чекбокс выключен, любые player-specific данные должны полностью игнорироваться.
- Если про-игрок не найден, система должна продолжать расчет только по героям.
- Если найденный про-игрок оказался `anonymous`, player-specific блок должен считаться недоступным и не ломать общий сценарий.
- Если игрок найден, но статистики на выбранном герое слишком мало, нужно либо снижать вес player-specific вклада, либо полностью уходить в hero-only fallback.
- Если STRATZ временно недоступен, форма и итоговый расчет не должны падать.
- Все fallback-сценарии должны быть детерминированными и одинаково работать для Radiant и Dire.
